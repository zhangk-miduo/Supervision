# 变更提案：增强企微目标、国家工作日、执行可观测性与组织视图

## 背景

当前任务只能选择一个群机器人配置，机器人管理把系统内部 `RobotId` 暴露为必填项，并用一个“名称”同时表达群和消息推送。实际业务中同一个企微群可以创建多个消息推送 Webhook，用户需要用“群名 + 消息推送名称”识别目标，并让一次任务触达到多个群。

当前调度已有 `holidayPolicy` 字段，但执行器仅按 Cron 触发，无法识别国务院法定节假日和周末调休工作日。执行日志只展示任务 ID 和原始文本/接口结果，且前端状态筛选值与后端成功、失败编码相反，业务用户难以判断实际推送了什么、哪些群成功或失败。

组织人员页面目前只有简单部门树和成员表，缺少同步概览、多维筛选、部门人数、成员详情和逐成员同步状态，无法达到附件参考界面的可用性。

## 目标

- 机器人管理拆分逻辑群与消息推送配置，同一群可维护多个推送。
- `RobotId` 改为系统生成的兼容标识，不再要求用户填写。
- 任务支持多选消息推送目标，并按目标独立发送、限流、幂等和记录结果。
- 增加“国家工作日”调度，正确处理法定放假和周末调休上班。
- 执行日志展示执行时的任务、消息和目标快照，以及业务可读的聚合结果和逐目标结果。
- 按附件结构升级组织人员管理：同步概览、筛选、组织树计数、成员/部门详情和同步日志入口。
- 继续保持企微人员与后台登录账号完全分离，人员不可手工新增。

## 非目标

- 不尝试通过群机器人 Webhook 自动读取群名称或群成员名单。
- 不把企微 `userid` 强制解释为企业工号。
- 不允许在组织页面编辑并回写企微基础资料。
- 不依赖非官方第三方节假日接口作为运行时唯一事实来源。
- 不在本变更中实现企微扫码登录或组织人员自动创建登录账号。

## 能力范围

1. `wecom-messaging`：群与推送分层、多目标扇出、逐目标幂等及聚合状态。
2. `advanced-scheduling`：国家法定工作日日历、预览、运行守卫和缺失告警。
3. `execution-observability`：任务/消息/目标快照、业务结果 DTO、错误翻译和详情时间线。
4. `wecom-directory-sync`：附件式组织管理布局、成员扩展字段、筛选、计数和同步可观测性。

## 影响范围

- 数据库：新增逻辑群、年度工作日日历与执行快照字段；扩展 Webhook、人员、部门和投递记录。
- 后端：消息扇出服务、工作日日历服务、执行查询聚合 DTO、组织分页查询和统计接口。
- 前端：机器人分层管理、多目标选择器、工作日调度、执行日志列表/详情、组织人员管理重构。
- 迁移：兼容旧 `robotId`、单个 `webhookId`、旧机器人名称和既有执行记录。
- 运维：年度日历数据更新、覆盖状态监控、部署前备份及上线后冒烟。

## 验收摘要

- 新建消息推送仅需群名、推送名称和 Webhook；系统标识自动生成。
- 同一群可维护多个推送，任务选择器显示“群名 · 推送名称”并支持多选。
- 三个目标两成功一失败时显示“部分成功 2/3”，重试不重复发送已成功目标。
- 法定节假日不执行，国务院安排的周末调休工作日正常执行，预览与实际一致。
- 年度日历缺失时系统不静默猜测，页面和运行日志均明确告警。
- 执行日志显示执行时任务名称、推送摘要和逐目标结果，不直接把接口响应作为主结果。
- 组织页面支持附件中的核心筛选、组织树人数、成员/部门详情和同步状态，同时没有手工新增人员入口。

## 依赖与风险

- 群名无法从 Webhook 可靠取得，必须由管理员维护，旧数据需要人工补齐。
- 多目标发送会引入部分成功、目标级重试和重复触达风险，必须按目标建立幂等键。
- 国务院年度通知没有已确认的稳定机器 API，需要维护经审核的年度数据包及来源元数据。
- 历史执行缺少快照，只能标记为“历史数据不完整”，不能伪造当时内容或群名称。
- OpenSpec CLI 当前环境不可用，本工件按既有 `spec-driven` 结构生成；安装 CLI 后需补做状态和格式校验。
