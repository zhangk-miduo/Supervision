# 变更提案：修复任务目标群选择

## 背景

机器人管理已存在启用机器人及对应的加密 Webhook 配置，但任务创建向导无法显示目标群。服务器日志表明 `/api/persons` 因旧版 `supervision_person` 表缺少 `tenant_key` 返回 500；前端又通过单个 `Promise.all` 同时加载人员和机器人，人员请求失败后机器人结果也不会写入页面状态。

V3 迁移使用 `CREATE TABLE IF NOT EXISTS` 创建组织表，无法升级已有旧表。这使数据库兼容缺陷最终以“目标群为空”的方式暴露，并造成两个本应独立的数据源互相阻断。

## 目标

- 用增量迁移将旧人员、部门及关系表安全对齐当前领域模型。
- 人员与目标群独立加载、独立报错，单一数据源故障不阻断另一项选择。
- 每次打开任务向导或进入消息触达步骤时刷新可选目标群。
- 只允许选择启用且存在有效加密 Webhook 配置的群机器人。
- 保持现有任务保存字段和发送链路兼容，不暴露 Webhook 明文。

## 非目标

- 不改变企微通讯录同步的业务规则。
- 不新增手工维护人员或部门能力。
- 不重做消息编辑器或机器人管理模块。
- 不在本变更中引入新的消息渠道。

## 影响范围

- 数据库：新增 Flyway 兼容迁移，修复既有组织表结构及索引。
- 后端：收敛“可选择目标群”的查询契约，并校验机器人与加密配置的启用状态。
- 前端：拆分人员与机器人加载状态，增加刷新、空态和错误提示。
- 测试与运维：增加旧库迁移测试、接口降级测试和部署后冒烟检查。

## 验收摘要

- 服务器既有数据库升级后，人员与部门查询不再因缺少字段返回 500。
- 即使人员接口失败，任务创建页仍展示可用目标群。
- 新增机器人后重新打开向导或进入消息触达步骤即可选择，无需整页刷新。
- 禁用或缺少有效加密配置的机器人不出现在选择列表中。
- 已保存任务的 `webhookId` 与实际发送使用的加密 Webhook 配置一致。

## 风险与约束

- 生产旧表的完整字段形态需在实施前以只读元数据查询确认，迁移必须可重复且不得丢失人员、部门和历史关联。
- 若旧、新标识字段同时存在，需要定义确定性回填规则并在添加非空或唯一约束前检测冲突。
- OpenSpec CLI 当前环境不可用，本工件按仓库现有 `spec-driven` 结构创建；安装 CLI 后需补做状态与格式校验。
