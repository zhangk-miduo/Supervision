# 设计文档：修复任务目标群选择

## 1. 问题链路

```text
任务向导加载
  └─ Promise.all(人员接口, 机器人接口)
       ├─ /api/persons → 旧表缺 tenant_key → HTTP 500
       └─ /api/robots → 返回可用机器人
            └─ Promise 整体拒绝，两个赋值都不执行
                 └─ 目标群下拉为空
```

机器人数据和加密 Webhook 数据已经存在且 ID 对齐，因此不能通过重复添加机器人解决。

## 2. 数据库兼容迁移

新增下一版本 Flyway 迁移，先读取并兼容生产旧表，而不是删除或重建：

1. 对 `supervision_person`、`supervision_department`、`supervision_person_department` 增加当前模型缺失的列。
2. 以默认租户键回填 `tenant_key`；从可用旧字段回填企微外部 ID 和展示字段。
3. 在设为非空、建立唯一索引前检查空值与重复值。
4. 保留原主键，避免破坏账号绑定、接收人快照和历史记录引用。
5. 迁移使用条件化 DDL 或存储过程方式保证既可作用于旧库，也可安全作用于已是新结构的库。

实施前必须通过 `information_schema.columns`、`statistics` 和外键元数据确认生产表实际结构，并据此完成精确映射。迁移失败时 Flyway 应阻止应用以半兼容结构启动。

## 3. 可选择目标群契约

短期保持任务模型使用 `webhookId`。后端提供或收敛一个选择器接口，只返回：

- 机器人/配置稳定 ID；
- 用户可识别的名称；
- 启用状态所需的非敏感展示信息。

查询必须以内含加密地址且启用的 `supervision_wecom_webhook` 为有效性事实来源；不得返回加密密文、完整 URL 或 key。为兼容现有机器人管理，可关联旧机器人名称，但不存在有效配置时不得进入选择器。

## 4. 前端加载与交互

人员和目标群使用独立异步状态：

```text
loadPersons() → people / peopleError / peopleLoading
loadTargetGroups() → groups / groupsError / groupsLoading
```

可使用 `Promise.allSettled` 并分别提交结果，或直接发起两个独立请求。不得因人员请求失败清空已经成功返回的群列表。

刷新时机：

- 打开“新建任务”向导；
- 编辑任务并进入消息触达步骤；
- 用户点击目标群旁的刷新动作。

交互要求：加载中显示加载状态；接口失败显示局部错误和重试；成功但为空时提示先配置并启用机器人；已保存但当前失效的目标需要明确标记，不能静默替换。

## 5. 保存与发送一致性

创建/更新任务时，后端重新验证 `webhookId` 对应配置存在、启用且可解密。执行时仍从加密配置表读取地址，禁止回退到旧表明文字段。配置在任务保存后被禁用时，执行形成可审计失败记录，而不是发送到其他群。

## 6. 验证策略

- 迁移测试：分别从旧表结构、新表结构升级，验证数据、索引、外键和幂等性。
- 后端测试：选择器过滤禁用/缺配置项；任务保存拒绝无效 `webhookId`；响应不泄露秘密。
- 前端测试：人员失败机器人成功、机器人失败人员成功、重新打开向导刷新、空态和重试。
- 服务器冒烟：Flyway 成功、`/api/persons` 200、目标群接口 200、创建任务保存目标群、测试发送命中同一配置。

## 7. 回滚与发布

数据库迁移仅新增和回填，不在同一版本删除旧列。应用回滚时旧版本仍可读取原字段；待至少一个稳定发布周期后再单独提出旧列清理。发布脚本需先备份数据库，部署后执行只读结构检查和接口冒烟。
