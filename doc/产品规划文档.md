## 1. 产品概述

### 1.1 产品定位
Supervision 是一个面向企业的**智能督办与自动化执行平台**。它将以往依赖人工记忆与人工推动的重复性督办事项（日报提醒、异常巡检、进度跟进、数据核对等），转化为由系统自动触发的执行流：

> **定时触发 → 数据查询 → 规则判断 → 消息通知 → 全程留痕**

### 1.2 产品理念

| 维度 | 传统模式 | Supervision 模式 |
| --- | --- | --- |
| 触发 | 人工检查 / 记忆 | 系统自动触发（定时 / 事件） |
| 查询 | 人工登录各系统核对 | HTTP / 数据节点自动查询 |
| 判断 | 人脑判断阈值与异常 | 条件节点规则判断（=、>、contains…） |
| 通知 | 人工在群里 @ 人提醒 | 企业微信机器人自动推送 |
| 跟进 | 易遗漏、滞后、无痕 | 执行日志全量记录、可追溯 |

**价值主张（Slogan）：** 让企业管理，从人工推动走向智能督办。

### 1.3 产品愿景
- **短期（V1.0）**：以企业微信为单一通知渠道，沉淀「任务 + 节点 + 调度 + 执行」的轻量自动化引擎。
- **中期（V2.0）**：扩展组织协同（组织同步 / @人 / 权限 / 多租户），从「单人督办」走向「组织级督办」。
- **长期（V3.0）**：引入 AI 能力（任务自动生成、风险预警、智能助手），进化为企业级督办中枢，并启用子品牌 **Supervision AI**。

## 2. 背景与问题

企业在日常运营中存在大量「需要定期做、但靠人记容易忘」的督办事项：

- 日报 / 周报是否按时提交？
- 线上系统是否出现异常？
- 关键业务指标是否突破阈值？
- 项目节点 / 巡检任务是否按期完成？

这些事项普遍具备特征：**高频、规则明确、依赖人工推动、遗漏成本高**。现有 PM / IM 工具难以把「查询—判断—通知」串成自动化闭环，Supervision 正是为此类场景而生。

## 3. 目标用户与角色

| 角色 | 说明 | 核心诉求 |
| --- | --- | --- |
| 督办管理员 | 配置督办任务与节点 | 低门槛搭建自动化流，无需写代码 |
| 业务负责人 | 接收并响应督办提醒 | 准时、准确、可追溯的提醒 |
| 团队 / 群成员 | 被 @ 或通知的对象 | 减少无效打扰，关键事项不遗漏 |
| 运维 / 开发 | 负责平台部署与维护 | 复用现有技术栈，运维简单 |

---

## 4. 典型应用场景

1. **日报 / 周报提醒**：17:30 定时触发 → 查询提交情况 → 未提交者企微提醒。
2. **系统异常自动巡检**：定时拉取监控/日志接口 → 判断异常 → 推送告警到运维群。
3. **业务数据阈值告警**：定时查询业务指标 → 超阈值则通知业务负责人。
4. **任务进度督办**：根据进度节点配置 → 到期未完自动提醒责任人。

## 5. 核心功能模块（MVP）

### 5.1 自动化任务管理（Task）
- 任务的创建、编辑、删除、启停。
- 每个任务由一组有序节点（Node）串联成执行流。
- 支持手动触发执行（调试 / 应急）。

### 5.2 流程节点（Node）
MVP 内置三类节点，按 `Trigger → HTTP → Condition → Notification` 编排：
- **HTTP 节点**：调用外部接口查询数据。
- **Condition 节点**：基于字段做规则判断。
- **Wechat 节点**：通过企业微信机器人发送通知。
- （远期扩展：SQL / Email / DingTalk / Feishu / AI 节点）

### 5.3 定时调度
- 基于 Cron 表达式的定时触发（如每天 17:30）。
- 调度状态管理（启用 / 停用）。

### 5.4 执行日志与监控
- 每次执行的记录（状态、结果、起止时间）。
- 执行历史可查询、可追溯。

### 5.5 企业微信机器人
- 机器人的增删改查与 Webhook 配置。
- 发送前可「测试」验证连通性。

## 7. 产品 Roadmap

### V1.0 — 企微督办（MVP）
- 任务 / 节点 / 调度 / 日志 / 机器人核心闭环。
- 单渠道（企业微信）。
- 管理后台（Vue3 + Element Plus）。
- Docker Compose 部署。

### V2.0 — 组织协同与多租户
- 组织同步（从企业微信同步部门 / 成员）。
- @人 精准提醒。
- 权限控制（角色 / 数据权限）。
- 多租户隔离。

### V3.0 — AI 智能督办
- AI 自动生成任务 / 节点配置。
- 风险预警（基于历史执行预测遗漏 / 异常）。
- 智能督办助手（对话式配置与查询）。
- 多渠道（邮件 / 钉钉 / 飞书）统一编排。

## 9. 成功指标（建议）

| 指标 | 说明 | 目标（示例） |
| --- | --- | --- |
| 督办覆盖率 | 已自动化的事项 / 应督办事项 | 持续提升 |
| 遗漏率 | 应提醒但未提醒的次数 | 趋近于 0 |
| 平均响应时延 | 事项发生到通知到达 | < 1 分钟（定时类按调度粒度） |
| 配置效率 | 新建一个督办流的平均耗时 | 非技术用户 < 10 分钟 |

## 10. 设计原则（产品侧）

- **简单**：不做重型 BPM，轻量节点编排即可覆盖绝大多数场景。
- **可扩展**：以 Node 策略模式为基础，新渠道 / 新节点即插即用。
- **企业级**：Docker 部署、审计留痕、权限可控，契合企业规范。
