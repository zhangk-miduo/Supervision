# 项目对话结果文档化规则

> **版本**：v1.0 | **日期**：2026-07-22 | **状态**：已落地

## 核心原则

凡涉及 Supervision 仓库的需求讨论、方案探索、代码分析、开发实施、缺陷排查、验证验收和决策变更，均须在本轮任务结束前把有效结果梳理到 `doc/` 项目文档中。聊天记录不是项目事实的唯一载体。

## 为什么需要此规则

- 防止需求、约束和设计决策只存在于单次聊天中。
- 让后续开发者和 Agent 能从仓库恢复完整上下文。
- 保证实现、验收和文档使用同一套结论。
- 对发生变化的结论保留明确、可追踪的项目记录。

## 规则条目

1. 每次与本仓库有关的对话完成前，必须判断本轮是否产生了值得沉淀的结果。
2. 应记录的内容包括：用户需求、分析结论、确认的决策、关键证据、影响模块、风险、未决问题、后续行动，以及实际完成的修改和验证结果。
3. 已有对应主题文档时优先更新原文档，避免为同一主题重复建文档。
4. 没有对应文档时，按照 `doc-writer` 约定创建 `spec.*`、`design.*`、`plan.*`、`rule.*`、`guide.*`、`report.*` 或 `debt.*` 文档。
5. 新文档文件名使用 `doc/{type}.supervision[.{module}].md`，模块名采用 kebab-case。
6. 文档必须包含版本、日期和状态，并以当前已确认结论为准，不机械复制聊天全文。
7. 后续对话推翻或修正旧结论时，必须在同一任务中更新对应文档，必要时记录变更原因。
8. 完成回复中应指出本轮更新了哪些项目文档。

## 判断标准

以下任一情况成立，即必须更新文档：

- 新增、修改或澄清了产品需求。
- 形成了架构、数据模型、接口、交互或实施决策。
- 发现了缺陷、技术债、风险或重要现状。
- 修改了代码、配置、部署、测试或运行方式。
- 得出了可复用的调查、排障或验收结论。

纯寒暄、与仓库无关的咨询，以及没有产生新信息的简单状态询问，可以不更新项目文档。

## 例外

- 不得为了满足本规则写入密码、密钥、Token、个人敏感数据或其他秘密信息。
- 尚未确认的推测必须标记为“待确认”，不能写成既定事实。
- 用户明确要求暂不落盘的敏感讨论，应停止写入并在最终回复中说明未文档化原因。
