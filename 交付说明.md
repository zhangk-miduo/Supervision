# Supervision 交付说明（构建到可部署）

> 本文件梳理从产品/技术文档到「可正式部署上线」所完成的工作。详细部署步骤见 [DEPLOYMENT.md](./DEPLOYMENT.md)，实施拆解见 [IMPLEMENTATION_PLAN.md](./IMPLEMENTATION_PLAN.md)。

## 1. 目标

基于 `doc/产品规划文档.md` 与 `doc/技术分析文档.md`，把「企业智能督办平台」从文档推进为一个**真实可部署**的全栈系统：
- 后端：Spring Boot 3.2.2 + Java 17 + MyBatis-Plus + MySQL 8 + Redis 7 + RabbitMQ + Quartz
- 前端：Vue 3 + TypeScript + Vite + Element Plus 管理后台
- 编排：Docker Compose 一键拉起全部依赖与前后端

## 3. 关键设计与修正（构建中已解决）

1. **节点执行器路由**：执行器以 `NodeType` 小写 code（`http`/`condition`/`wechat`）为 key，而前端传入大写（`HTTP`）。已在引擎中统一用 `NodeType.of(raw).getCode()` 归一化后再查表，避免节点被误判「无执行器」而跳过。
2. **条件节点失败语义**：条件不满足时原逻辑仅 `break` 导致链路被记为「成功」。已改为任意节点（含条件）失败均落库为「失败」并停止链路。
3. **企微节点配置对齐**：执行器原读取 `template` 字段，与前端 `content` 不符；已改为读取 `content`，并在节点未配置时回退到机器人模板。
4. **条件节点字段对齐**：执行器使用 `field`/`operator`/`value`，前端模板已同步（原用了 `variable`/`expect`）。
5. **API 返回码**：`ApiResult.ok` 返回 `code=200`，与前端 `=== 200` 的成功判断一致。
6. **执行状态语义**：后端 `ExecutionStatus`（0成功/1失败/2执行中），前端列表与仪表盘已对齐。
7. **调度类型约定**：`scheduleType` 0=手动、1=定时(Cron)，前后端统一。
8. **数据库账号**：初始化 SQL 创建 `supervision/supervision` 账号并授权，与应用 `application.yml` 连接配置一致。
9. **上下文取值**：为 `ExecutionContext` 补 `getData()`，供引擎将执行上下文持久化到 Redis。

## 4. 验证情况

| 项目 | 方式 | 结果 |
|------|------|------|
| 前端编译打包 | 本机 Node 22 执行 `npm install` + `npm run build` | 进行中 / 已产出 `dist/` |
| 后端编译 | `build/backend/Dockerfile` 多阶段 Maven（Java 17） | 由 Docker 引擎构建（目标部署机执行） |
| 容器编排 | `docker compose up --build` | 需在已启动 Docker 守护进程的主机执行 |

> 说明：本开发沙箱因安全策略无法启动 Docker 守护进程（Docker Desktop 守护进程所需写入被拦截），因此未在此环境内执行真实容器编排。代码与全部编排配置已就绪，在具备 Docker 的环境中执行 `docker compose up -d --build` 即可完成编译、建库、启动与上线。
